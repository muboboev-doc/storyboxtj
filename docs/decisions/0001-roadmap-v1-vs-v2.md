# ADR 0001: Roadmap v1 → v2 (vibe-coding оптимизация)

**Статус:** принят
**Дата:** 2026-04-29
**Контекст:** CLAUDE.md раздел 17

---

## Контекст

Первый roadmap (v1) в CLAUDE.md был построен по классической модели — 9 этапов горизонтальными срезами:

```
Этап 0  — Подготовка (1 нед)
Этап 1  — Backend Core (3 нед)
Этап 2  — Mobile MVP (4 нед)
Этап 3  — IAP (3 нед)
Этап 4  — Banks (3 нед)
Этап 5  — Retention (2 нед)
Этап 6  — Web (2 нед)
Этап 7  — Admin polish (1 нед)
Этап 8  — Anti-piracy (2-3 нед)
Этап 9  — Release (2 нед)
Итого:    23-25 недель
```

Этот план хорошо описывает **что делать**, но не учитывает **как** Claude Code наиболее эффективен.

## Проблемы v1 для вайбкодинга

1. **Этапы-монолиты.** «Backend Core 3 недели» = 30+ Claude-сессий в одном бакете. Прогресс не виден промежуточно.
2. **Горизонтальные срезы.** Mobile не работает 7 недель. Нет рабочей демки до этапа 2 = негде верифицировать.
3. **Нет walking skeleton в день 1.** До этапа 4 нет полного e2e. Треки нельзя параллелить.
4. **Тесты после кода.** Для wallet / banks / IAP / DRM это деал-брейкер: тест — самый чёткий «критерий готовности».
5. **Codegen упомянут, но не центральная ось.** Filament resource генерится из миграции; Riverpod/Retrofit — из аннотаций; OpenAPI → клиент. Каждый сгенерированный файл = код, который Claude **не пишет**.
6. **Type contracts ручные.** API эндпоинт = контроллер + ресурс + Flutter-клиент → 3 места правки на каждое поле. OpenAPI + codegen = одно место.
7. **Reader-app iOS на этапе 4.** Архитектурное решение — должно быть на этапе 0.
8. **3 банка параллельно.** Первый банк = разведка боем. Второй и третий — копи-паст. Параллельно = 3× риска.
9. **Все 30 миграций на этапе 1.** БД должна эволюционировать с фичей.
10. **Нет demo milestone каждую неделю.** Без кликабельной демки вайбкодинг превращается в работу-ради-работы.
11. **Plan Mode не маркирован.** Wallet, banks, DRM требуют Opus + Plan mode. CRUD — Sonnet.

## Решение

Перейти на v2 — план оптимизированный под возможности Claude Code:

### Принципы v2

- **Walking Skeleton First** — фаза 0 за неделю даёт полный e2e stub, дальше всё нанизывается.
- **Vertical Slice** — одна фича = миграция + API + UI + тест в одном PR.
- **TDD для High-Risk** — wallet, banks, IAP, DRM начинаются с теста (red → green).
- **Type Contracts First** — OpenAPI + codegen клиента.
- **Codegen Maximization** — `riverpod_generator`, `retrofit_generator`, `freezed`, Filament `--generate`.
- **Mock-first для Parallelism** — backend пишет фейк → mobile параллелит.
- **One Session = One PR** — каждая задача атомарна, 30-90 минут.
- **Plan Mode + Opus для Risky** — явно отмечено в плане.
- **Demo Friday** — staging deploy каждую неделю.
- **Spike перед интеграцией** — для каждого банка сначала Postman-коллекция + ADR.
- **One-then-Template** — первый банк end-to-end до зелёного, потом остальные за 3 дня.

### Структура v2

```
Фаза 0   — Walking Skeleton (1 нед, ~10 сессий)
Фаза 1   — Auth Vertical Slice (1 нед, ~8 сессий)
Фаза 2   — Content Vertical Slice (1.5 нед, ~12 сессий)
Фаза 3   — Wallet + Unlock (TDD) (1 нед, ~8 сессий)
Фаза 4   — FFmpeg + Real Video (3-4 дня, ~5 сессий)
Фаза 5   — IAP (TDD) (1.5 нед, ~10 сессий)
Фаза 6   — Banks (one-then-template, 3 нед)
            6A spike Alif → 6B Strategy + Alif end-to-end → 6C UI Alif → 6D Eskhata → 6E DCBank → 6F demo
Фаза 7   — Retention (параллельные slice'ы, 1.5 нед)
Фаза 8   — Localization (3-4 дня, ~6 сессий)
Фаза 9   — Anti-Piracy (TDD) (2 нед)
Фаза 10  — Web (1 нед, ~8 сессий)
Фаза 11  — Admin polish (3-4 дня, ~5 сессий)
Фаза 12  — Release prep (1 нед)
Итого:    ~14 недель при том же качестве
```

## Последствия

**Плюсы:**
- Срок MVP сокращается с 23-25 до ~14 недель (~40% быстрее).
- Кликабельный продукт уже после фазы 0 (1 неделя), не после этапа 2 (8 недель).
- Каждая задача = одна Claude-сессия → видимый прогресс, чистая история коммитов.
- Параллелизм после walking skeleton — backend и mobile могут идти одновременно.
- TDD на high-risk модулях даёт чёткие критерии готовности.
- Reader-app iOS заложен с дня 1 — нет архитектурного refactor'а на этапе 4.
- Один банк end-to-end снимает 80% неизвестности перед двумя следующими.

**Минусы / риски:**
- Vertical slice требует контекст-переключений между backend/mobile в одном PR — для одного разработчика это окей, для команды — может тормозить.
- TDD ускоряет качество, но замедляет первые 30 минут задачи (написать тест перед кодом).
- Плотные таблицы в roadmap менее читаемы, чем чек-листы — компенсируется детальными DoD на каждую фазу.
- Codegen-зависимость означает периодические `flutter pub run build_runner build` — нужно уметь дебажить codegen-failures.

## Альтернативы, которые рассматривались

1. **Оставить v1.** Отвергнуто — слишком монолитные этапы, прогресс не виден, нет демок.
2. **Гибрид (v1 структура + TDD маркеры).** Отвергнуто — не решает проблему горизонтальных срезов и monolith-этапов.
3. **Полный agile с двухнедельными спринтами.** Отвергнуто — для одного разработчика + Claude формальная агильная церемония избыточна. Demo Friday + Plan Mode перед high-risk — достаточно.
